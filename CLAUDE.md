# CLAUDE.md ‚Äî Fantasy Football League Platform (v2)

## Project Overview

A multi-tenant SaaS web app for tracking fantasy football league stats across seasons.
Commissioners create a league, enter teams/weeks/scores; members view standings, all-time
records, head-to-head matchups, owner career stats, and playoff brackets.

**Backend:** PHP/Laravel microservices monorepo, CQRS + Event-Driven Architecture (EDA),
generated by the **LAF (Laravel Auto-Framework)** agent
(`/Users/ryanwinn/Documents/Developments/Synergy/laf-agent`) from a DB schema definition.
LAF-generated bugs most often stem from wrong table names or bad data-type definitions in the
schema step ‚Äî see "Known LAF Generation Bugs" below.

**Reference backend (proven, ~working):**
`/Users/ryanwinn/Documents/Developments/Synergy/synergycore/tenant-api-net` ‚Äî v2 reuses its
CQRS + EDA, schema-per-tenant pattern exactly. Check it first when in doubt about routing,
docker setup, or SQL schema structure. **League = tenant** is the only deliberate divergence in
the domain model.

**Frontend:** React + TypeScript + Vite + Tailwind, mirroring `tenant-web`.

**v1 (port source):** `/Users/ryanwinn/fantasy-football-web` ‚Äî single-league React+Express app
backed by one flat `seasons.json`, no auth. Components are solid and worth porting; the data
layer and lack of auth are exactly what v2 replaces.

### Route Convention

All routes follow a strict CQRS-style POST pattern:

```
POST /{service-name}/{model-name}/{action}
```

Actions per model: `create`, `update`, `delete`, `restore`, `duplicate`, `batch-update`,
`batch-delete`, `fetch` (by ID), `search` (filter/query).

Example: `POST /league-service/team/search`

**Auth endpoints (special cases):**
- `POST /auth-service/login`
- `POST /auth-service/logout`
- `POST /auth-service/refresh`

The nginx gateway strips the `/{service-name}/` prefix; Laravel only ever sees `->prefix('api')`.

## Tenancy Model

**League = tenant.** Each league has its own MySQL schema. The JWT `schema` claim selects it on
every request.

- The `schema` claim is set from `accounts.tenants.schema` in auth-service; every other service
  reads it to pick the correct tenant database.
- **Never trust the claim alone** ‚Äî validate it against the principal's league membership on every
  request, so a leaked/forged claim can't cross leagues.
- **Onboarding:** a new league must have its schema created in *every* service's MySQL and infra
  tables seeded (`inbox_items`/`outbox_items`, caches) **before** the first JWT is issued.
  Automate via the orchestrator onboarding saga (see Bug 7/9).

## Authorization (roles, scoped per league)

- **Commissioner** ‚Äî full CRUD on their league (teams, weeks, scores, settings, invites).
- **Member** ‚Äî read all; edit own team metadata only.
- **Viewer / public** ‚Äî read-only public league pages (opt-in per league).

Enforce with route middleware. Every non-public route is behind auth.

## Known LAF Generation Bugs

These are ported from `tenant-api-net/CLAUDE.md` (the same generator produced both projects, so
**these bugs WILL recur**). Service names below are re-contextualized to fantasy-app. Run
`/laf-audit` immediately after every LAF generation pass.

### Bug 4: Query Contracts Extend Non-Existent `BaseQueryContract` (affects ALL modules)

LAF generates query contracts (`Search*`, `Fetch*`) extending a `BaseQueryContract` that does not
exist in the shared library.

**Symptom:** `Class "...\Contracts\Queries\BaseQueryContract" not found` on any search/fetch.

**Fix:** Correct base classes (in `shared/src/Contracts/Queries/`):
- `Search*QueryContract` ‚Üí extends `SearchQueryContract`
- `Fetch*QueryContract` ‚Üí extends `FetchQueryContract`

**Audit:**
```bash
grep -r "BaseQueryContract" {module}/src/app/Contracts --include="*.php" -l | grep -v vendor
```

### Bug 8: Search/Fetch Override Contracts Have Wrong `rules()` (affects ALL modules)

LAF gives every `Search*`/`Fetch*` contract a `rules()` override with create-style field
validation (`'label' => 'required|...'`), forcing search/fetch to supply all model fields.
Additionally `ContractFactory::createOverrideContract()` calls override contracts with the wrong
constructor args.

**Symptom:** `TypeError: SearchQueryContract::__construct(): Argument #3 ($table) must be of type
string, OperationKeyContext given`; or search/fetch demanding every field.

**Fix (two parts):**
1. `shared/src/Controllers/Execution/ContractFactory.php` ‚Äî detect `SearchQueryContract` /
   `FetchQueryContract` subclasses and call with correct signatures:
   - `Search*`: `($requestData, $tenantSchema, $model->getTable(), ['id', ...$model->getFillable()], $operationKeyContext)`
   - `Fetch*`: `($requestData, $tenantSchema, $model->getTable(), $operationKeyContext)`
2. Remove the `rules()` override from every `Search*`/`Fetch*` contract ‚Äî the base class rules are
   correct.

**Audit:**
```bash
grep -rl "extends SearchQueryContract\|extends FetchQueryContract" {module}/src/app/Contracts --include="*.php" | grep -v vendor
# each should have NO rules() override
```

### Bug 10: Command Contracts Have Broken `rules()` Overrides (affects ALL modules)

LAF generates `Create*`/`Update*`/`Delete*` command contracts with hardcoded `rules()`:
1. **HTML-entity encoding** ‚Äî `exists` rules generated with `&#x27;`/`&gt;` instead of raw PHP,
   producing broken SQL.
2. **Wrong delete rules** ‚Äî delete contracts require all model fields when only the identifier is
   needed.

**Root cause:** `BaseCommandContract::rules()` already generates correct manifest-based validation.
The LAF overrides are redundant AND broken.

**Fix:** Remove `rules()` from all command contracts. **Preserve** `afterValidation()` methods ‚Äî
those hold legitimate custom logic.

### Bug 1: All Services Default to Port 9000 (affects ALL modules)

LAF generates every service with `--port=9000` and `{n}:9000`, but the nginx `registrar.json`
routes each service to a unique port. Container-to-container networking uses the internal port, so
each service must listen on its registrar port.

**Symptom:** 502 Bad Gateway on every request through the gateway.

**Fix:** In each module's `docker-compose.development.yml`:
```yaml
command: php artisan serve --host=0.0.0.0 --port=<registrar-port>
ports:
  - "<registrar-port>:<registrar-port>"
```
Assign unique ports per service in `infrastructure/gateway/registrar.json` (e.g. auth-service
9001, permission-service 9002, league-service, matchup-service, standings-service each distinct).

### Bug 2: Outbox Worker Missing Entrypoint Mount (affects ALL modules)

The development Dockerfile stage never copies `entrypoint-outbox.sh` (only production does), and
the compose template omits the volume-mount workaround it correctly applies to inbox workers.

**Symptom:** `OCI runtime create failed: exec: "/usr/local/bin/entrypoint-outbox.sh": no such file
or directory`

**Fix:** Add to every outbox worker's `volumes:` in `docker-compose.development.yml`:
```yaml
- ./{service}/docker/entrypoint-outbox.sh:/usr/local/bin/entrypoint-outbox.sh
```

### Bug 3: `db_users.sql` Hardcodes Password via DROP/CREATE USER

LAF generates `db_users.sql` with `DROP USER / CREATE USER ... IDENTIFIED ... BY '...'`, hardcoding
a password that won't match `MYSQL_PASSWORD`.

**Symptom:** `SQLSTATE[HY000] [1045] Access denied for user 'laravel'@'%'`

**Fix:** `db_users.sql` must **GRANT only ‚Äî never create the user**. Docker Compose's `MYSQL_USER`
/ `MYSQL_PASSWORD` create the user with the right password.
```sql
GRANT SELECT, INSERT, UPDATE, CREATE, INDEX, SHOW DATABASES ON *.* TO 'laravel'@'%';
FLUSH PRIVILEGES;
```

### Bug 9: LAF Does Not Generate `service_infra_models.sql` (affects ALL non-auth modules)

LAF generates `models.sql` (domain tables + `transaction_events` in the `tenant` schema) but never
creates `inbox_items`/`outbox_items` in the service's own schema. Without them the outbox worker
crashes on every transaction.

**Symptom:** `Table '{service}.outbox_items' doesn't exist`

**Fix:** Create `service_infra_models.sql` in each service's `docker/mysql-init/`:
```sql
USE `{service-name}`;
SET FOREIGN_KEY_CHECKS=0;
CREATE TABLE IF NOT EXISTS `inbox_items` ( ... );
CREATE TABLE IF NOT EXISTS `outbox_items` ( ... );
SET FOREIGN_KEY_CHECKS=1;
```
Use `CREATE TABLE IF NOT EXISTS` (non-destructive). Since MySQL init runs only on fresh volumes,
apply manually to a running container:
```bash
docker exec -i {service}-db mysql -uroot -p'ROOT_PASSWORD' < docker/mysql-init/service_infra_models.sql
```

### Bug 7: JWT `schema` Claim Must Match an Existing Tenant DB (by-design tenancy, not a bug)

The `schema` claim is set from `accounts.tenants.schema`. Every service reads it to select the
tenant DB. If that DB doesn't exist in a service's MySQL, the request fails.

**Symptom:** `Schema '{name}' does not exist. Client may not be properly onboarded.`

**Dev fix:** Ensure `accounts.tenants.schema` matches the DB created in each service's
`db_schemas.sql`, then re-issue a JWT (login again).
```sql
UPDATE accounts.tenants SET `schema` = '{league-schema}' WHERE id = {tenant_id};
```
**Production:** onboard via the orchestrator saga ‚Äî create the schema in every service before
issuing the league's first JWT.

### Bug 5: Frontend-Generated Types Use Wrong Route Prefix (affects ALL modules)

The frontend's auto-generated `{Model}Api` URL paths may not match the backend route prefix in
`routes/api.php` ‚Äî LAF names backend routes from the PHP model name, the frontend generator may
differ.

**Fix:** Cross-check every model's `{Model}Api.search` path against `Route::prefix(...)` in the
service's `routes/api.php`. They must match exactly.

### Bug 6: Frontend Auth Interceptors Disabled in Mock Mode (frontend only)

`src/lib/apiClient.ts` ships with both the request interceptor (JWT attach, idempotency key) and
the response interceptor (401 refresh) commented out behind a `/* MOCK MODE */` block.

**Symptom:** No `Authorization` header sent; every request fails with "No schema in token."

**Fix:** Uncomment both interceptors and set `VITE_USE_MOCK_DATA=false` in the frontend `.env`.

### Bug 11: Frontend-Generated Type Has Wrong Column Names (frontend)

LAF's frontend type generator uses conceptual model names instead of actual DB column names (e.g.
`bid_round_id` vs `bid_phase_id`, `estimate_id` vs `estimate_puid`).

**Symptom:** `422 ‚Äî The {field} field is required` on every create; records never load (search
filter silently returns nothing because the column doesn't exist).

**Fix:** The backend `manifest.json` is authoritative. Rename fields in the generated `.ts`
(interface, Zod schemas, `{Model}Fields` metadata) and update all consumers. Watch this when
porting v1's domain fields onto generated types.

## Database Setup

**Do NOT use Laravel migrations.** Multi-tenant ‚Äî each league has its own MySQL schema. Setup is
via raw SQL init files mounted into each service's MySQL container at startup.

Each service's `docker/mysql-init/` contains:
- `db_schemas.sql` ‚Äî `CREATE DATABASE` for the service DB and each tenant schema
- `db_users.sql` ‚Äî DB user grants (**GRANT only ‚Äî never DROP/CREATE USER**, Bug 3)
- `service_infra_models.sql` ‚Äî `USE {service}` ‚Äî inbox_items / outbox_items (Bug 9)
- `service_tenant_models.sql` ‚Äî `USE tenant` ‚Äî shared infra (transaction_events, caches)
- `tenant_models.sql` ‚Äî domain tables, one `USE <tenant>` + `CREATE TABLE` block per tenant

**Auth-service is special** ‚Äî three schemas: `auth-service` (infra), `accounts` (cross-league auth
data), `tenant` (per-tenant infra cache).

**MySQL init scripts only run on fresh volumes.** If a volume already has data, init is silently
skipped. Apply manually:
```bash
docker exec -i {container} mysql -uroot -p'ROOT_PASSWORD' < docker/mysql-init/script.sql
# fully fresh (destructive): docker compose -f docker-compose.development.yml down -v && ... up
```

### SQL Schema Conventions
- `public_id` ‚Üí `CHAR(26) NOT NULL UNIQUE` (ULID)
- `public_ref` ‚Üí `VARCHAR(26) NOT NULL UNIQUE`
- `created_by_principal` ‚Üí `CHAR(26) NOT NULL`
- `version` ‚Üí `INT NOT NULL` (no DB default ‚Äî set by app)
- Cross-service references ‚Üí `CHAR(26) NULL` (`*_puid`)
- Internal FK references ‚Üí `BIGINT` (`*_id`)

## MCP Servers & Workflow

Project-scoped MCP servers are declared in committed `.mcp.json`. Credentials are **never**
committed ‚Äî they expand from the shell environment when `claude` launches.

### Configured servers

| Server | Purpose | Access |
|--------|---------|--------|
| `mysql` | Inspect tenant/service schemas directly (table layout, column types, sample rows) | **Read-only** ‚Äî `ALLOW_INSERT/UPDATE/DELETE_OPERATION=false`. Connect as the `laravel` user, never root. |
| `github` | Read issues/PRs/repo metadata | Optional ‚Äî disabled unless `GITHUB_PERSONAL_ACCESS_TOKEN` is set. We otherwise use the `gh` CLI. |

### Environment setup

Set these in your shell before launching `claude` (e.g. `~/.zshrc`, or
`set -a; source .mcp.env; set +a; claude`). Put real values in `.mcp.env` ‚Äî it is gitignored.

```bash
# mysql (read-only) ‚Äî point PORT at the per-service MySQL container you want to inspect;
# each service exposes its own host port in docker-compose.development.yml
FANTASY_MCP_MYSQL_HOST=127.0.0.1
FANTASY_MCP_MYSQL_PORT=3306         # e.g. the auth-service or league-service DB host port
FANTASY_MCP_MYSQL_USER=laravel
FANTASY_MCP_MYSQL_PASS=<dev password from that service's docker-compose>
FANTASY_MCP_MYSQL_DB=<schema to inspect, e.g. a tenant/league schema>

# github (optional) ‚Äî fine-grained PAT, read-only on this repo; leave unset to disable
GITHUB_PERSONAL_ACCESS_TOKEN=<pat>
```

The `mysql` server is the fastest way to confirm a manifest's field names/types against the actual
DB (Bug 11) and to verify a tenant schema exists before issuing a JWT (Bug 7). Treat
`manifest.json` as ground truth ‚Äî use the DB to *confirm*, not to override it.

### Plan-mode-first & audit discipline

- **Plan mode first** for any multi-file change (LAF fixes, porting a v1 screen, cross-service
  edits): enter plan mode, get the plan approved, then execute. Skip it only for trivial,
  single-file edits.
- **Run `/laf-audit` immediately after every LAF generation pass**, before touching the generated
  code ‚Äî it sweeps the known-bug catalog (`BaseQueryContract`, stray `rules()`, port 9000, missing
  `service_infra_models.sql`, `db_users.sql` DROP/CREATE USER).
- Run `/security-check` before any commit that adds routes or touches auth.

## Code Style & Conventions

- Models: `final class`, explicit `protected $table`, explicit `$fillable`, typed `$casts`.
- No `updated_at` on most models ‚Äî `const UPDATED_AT = null;`
- Soft deletes on all domain models; optimistic locking via `version`.
- Internal IDs are ints; external references use `public_id` (ULID) + `public_ref`.
- Comments only where the WHY is non-obvious. Don't add error handling for impossible scenarios ‚Äî
  trust the framework.

## Security Notes

- **NEVER commit `.env` files** or any secrets (API keys, tokens, JWT secrets). Verify `.env` is in
  `.gitignore` before every commit. (A commit-blocking secret-scan hook is added in Phase 0.3.)
- AuthN: JWT via `tymon/jwt-auth` (access + refresh) + Sanctum for API tokens. JWT carries the
  tenant `schema` claim.
- Every non-public route behind auth middleware; validate the `schema` claim against league
  membership (don't trust it alone).
- HTTPS only; secure/HttpOnly/SameSite cookies; CORS locked to the known frontend origin; rate
  limiting on auth + write endpoints; CSRF on cookie-auth flows; bcrypt/argon2 hashing.
- RabbitMQ credentials in docker-compose are dev-only ‚Äî never reuse in production.
- Audit trail via `transaction_events`; dependency scanning via Dependabot.
