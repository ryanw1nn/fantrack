# Fantasy Football League Platform — v2 Project Outline

## Context

**What:** A multi-tenant SaaS web app for tracking fantasy football league stats across
seasons. Commissioners create a league, enter teams/weeks/scores, and members view
standings, all-time records, head-to-head matchups, player (owner) career stats, and
playoff brackets.

**Why v2:** v1 (`/Users/ryanwinn/fantasy-football-web`) is a single-league React+Express app
backed by one flat `seasons.json`, no auth, standings recomputed on every read. It works but
is hardcoded to "The Fan Club" league. v2 generalizes this so **any user can run any league**,
with real authentication, authorization, tenant isolation, and production-grade security.

**How:** Backend is generated by the **LAF (Laravel Auto-Framework)** agent
(`/Users/ryanwinn/Documents/Developments/Synergy/laf-agent`) — the same toolchain that
produced `tenant-api-net`
(`/Users/ryanwinn/Documents/Developments/Synergy/synergycore/tenant-api-net`). v2 reuses that
proven CQRS + Event-Driven, schema-per-tenant microservices pattern. Frontend mirrors the
`tenant-web` React + TypeScript stack.

**Confirmed decisions:**

| Decision | Choice |
|---|---|
| Tenancy | **League = tenant (own MySQL schema)** — mirrors LAF isolation exactly |
| Data entry | **Manual + CSV import at launch**; **Sleeper sync** as first post-launch add |
| Frontend | **React + TypeScript**, mirroring `tenant-web` (generated clients/types, Vite) |
| Hosting | **Decide later** — plan for containerized deploy generically (Docker Compose now, AWS-ready) |

**Working directory:** `/Users/ryanwinn/Documents/Developments/FantasyApp` (currently empty).

## Architecture Overview

Reuse the `tenant-api-net` shape exactly. **League = tenant** ⇒ each league has its own MySQL
schema; the JWT `schema` claim selects it on every request (LAF "Bug 7" is by-design tenancy).

```
fantasy-app/
├── core-module/
│   ├── auth-service/          # Principals, Users, Tenants(=Leagues), JWT, login/logout/refresh
│   └── permission-service/    # Roles & permissions (commissioner / member / viewer)
├── league-module/
│   ├── league-service/        # League, Season, Team(owner), TeamSeason membership
│   ├── matchup-service/       # Week, Matchup, Score  (the v1 "weeks" model)
│   └── standings-service/     # Computed standings, playoff brackets, all-time aggregates
├── import-module/             # (post-launch) CSV/JSON import + Sleeper sync workers
├── orchestrator/              # Sagas: recompute-standings-on-score-change, onboarding
├── infrastructure/            # nginx gateway + registrar.json, RabbitMQ, adminer, AWS configs
└── shared/                    # TenantModel base, CQRS/EDA contracts (from LAF)
```

**Conventions inherited from LAF (do not reinvent):**
- Route pattern: `POST /{service}/{model}/{action}` where action ∈
  `create|update|delete|restore|duplicate|batch-update|batch-delete|fetch|search`.
- Special auth routes: `POST /auth-service/login|logout|refresh`.
- `TenantModel` base fields: `id`, `public_id` (ULID, `CHAR(26)`), `public_ref`, `label`,
  `created_at`, soft-delete `deleted_at`, `created_by_principal`, `version` (optimistic lock).
- Each service: own MySQL container, inbox worker (RabbitMQ consumer), outbox worker (event
  publisher). nginx gateway strips the `/{service}/` prefix (Laravel uses `->prefix('api')`).
- **`manifest.json` per service is ground truth** for field names/types — frontend types must
  match it exactly.
- **No Laravel migrations** — schema via raw SQL init files in `docker/mysql-init/`.

## Domain Model (generalized from v1)

v1's `seasons.json` shape (`teams[]` with owner `name`, `state`, `playoff{made,rounds,pChampion}`,
`rChampion`; `weeks{}` → `matchups[]` with `team1/team1Score/...`, `status`; derived `standings`)
maps onto these tenant-scoped models:

| Service | Models | Notes |
|---|---|---|
| **league-service** | `League`, `Season`, `Team`, `TeamSeason` | `Team` = a franchise; an owner (`Principal`) can hold a team across seasons → enables "player/owner career stats" (v1 `PlayerStatsPage`). `TeamSeason` carries `state` (active/inactive/eliminated), `playoff_made`, `playoff_rounds`, `is_playoff_champion`, `is_regular_champion`. |
| **matchup-service** | `Week`, `Matchup` | `Matchup` = `team1_id`, `team1_score`, `team2_id`, `team2_score`, `label`, `status` (regular/playoff/toilet/out). BYE handling preserved. |
| **standings-service** | `Standing` (materialized), `PlayoffBracket` | Replaces v1's recompute-on-read (`recalculateStandings` in `server.js`). Recomputed via an **orchestrator saga** triggered by matchup score events, then stored — avoids v1's per-request recompute cost and concurrency bugs. |

Carry over v1 metrics: W/L/T, PF, PA, PFPG, PAPG, win %, place + prevPlace (rank delta Δ),
playoff buckets (playoff/toilet/out), champions. Fix v1 bugs in the port (e.g. `server.js:154`
`'name2' === 'BYE'` string-literal bug; magic "week ≥ 15 = playoff" → make playoff weeks
league-configurable).

## Authentication, Authorization & Security

This is the headline upgrade over v1 (which had none).

- **AuthN:** JWT via `tymon/jwt-auth` (access + refresh) + Sanctum for API tokens — same as
  `tenant-api-net`. `login/logout/refresh` endpoints. JWT carries the tenant `schema` claim.
- **AuthZ:** `permission-service` roles scoped per league:
  - **Commissioner** — full CRUD on their league (teams, weeks, scores, settings, invites).
  - **Member** — read all, edit own team metadata only.
  - **Viewer / public** — read-only public league pages (opt-in per league).
  - Enforce with route middleware; every non-public route behind auth.
- **Tenant isolation:** schema-per-league means a leaked query can't cross leagues; verify the
  JWT `schema` claim is validated against the principal's league membership on every request
  (don't trust the claim alone).
- **Onboarding:** new league ⇒ create its schema in every service's MySQL + seed infra tables
  (`inbox_items`/`outbox_items`, transaction cache) before issuing the first JWT (LAF "Bug 7/9").
  Automate via an orchestrator onboarding saga so it's not manual.
- **Hardening checklist:** HTTPS only; secure/HttpOnly/SameSite cookies; CORS locked to the
  known frontend origin (v1 `server.js:18` already scoped — keep that discipline); rate
  limiting on auth + write endpoints; input validation via LAF command contracts (strip the
  broken generated `rules()` — LAF Bug 8/10 — base contracts validate from manifest); CSRF on
  cookie-auth flows; bcrypt/argon2 password hashing; audit log via the existing
  `transaction_events` / event store; dependency scanning (Dependabot); secret management via
  env + the commit-blocking hook from Phase 0.

## Platform Sync — Research Findings (post-launch feature)

Verdict: **ship manual + CSV first, add Sleeper next.**

| Platform | Difficulty | Auth | Notes |
|---|---|---|---|
| **Sleeper** | **Easy** | None | Free read-only REST at `https://api.sleeper.app/v1/`, ~90 req/min/IP. Endpoints for leagues, rosters, users, matchups, playoffs, transactions, drafts, players. **Do this one first.** |
| **ESPN** | Medium | `espn_s2` + `SWID` cookies (private leagues) | Unofficial v3 API. Public leagues need no auth; private leagues need the user to paste two cookies (cannot be obtained programmatically). Mature JS/Python libs exist. |
| **Yahoo** | Hard | OAuth2 (3-legged) + refresh | Official API but requires app registration and full OAuth token lifecycle. Defer. |

**Plan:** Build an `import-module` with a pluggable `LeagueImporter` interface. Implement
`CsvImporter` + `SleeperImporter` first; the importer maps external rosters/matchups onto the
domain models above and runs as an outbox/inbox worker so a large import doesn't block the API.
ESPN/Yahoo are later adapters behind the same interface.

**Sources:**
- [Sleeper API docs](https://docs.sleeper.com/)
- [ESPN private-league auth (ffscrapr)](https://ffscrapr.ffverse.com/articles/espn_authentication.html)
- [ESPN_S2 / SWID discussion](https://github.com/cwendt94/espn-api/discussions/150)
