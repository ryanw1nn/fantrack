# Fantasy Football League Platform — v2 Project Outline

## Context

**What:** A multi-tenant SaaS web app for tracking fantasy football league stats across
seasons. Commissioners create a league, enter teams/weeks/scores, and members view
standings, all-time records, head-to-head matchups, player (owner) career stats, and
playoff brackets.

**Why v2:** v1 (`/Users/ryanwinn/fantasy-football-web`) is a single-league React+Express app
backed by one flat `seasons.json`, no auth, standings recomputed on every read. It works but
is hardcoded to "The Fan Club" league. v2 generalizes this so **any user can run any league**,
with real authentication, authorization, tenant isolation, and production-grade security.

**How:** Backend is generated by the **LAF (Laravel Auto-Framework)** agent
(`/Users/ryanwinn/Documents/Developments/Synergy/laf-agent`) — the same toolchain that
produced `tenant-api-net`
(`/Users/ryanwinn/Documents/Developments/Synergy/synergycore/tenant-api-net`). v2 reuses that
proven CQRS + Event-Driven, schema-per-tenant microservices pattern. Frontend mirrors the
`tenant-web` React + TypeScript stack.

**Confirmed decisions:**

| Decision | Choice |
|---|---|
| Tenancy | **League = tenant (own MySQL schema)** — mirrors LAF isolation exactly |
| Data entry | **Manual + CSV import at launch**; **Sleeper sync** as first post-launch add |
| Frontend | **React + TypeScript**, mirroring `tenant-web` (generated clients/types, Vite) |
| Hosting | **Decide later** — plan for containerized deploy generically (Docker Compose now, AWS-ready) |

**Working directory:** `/Users/ryanwinn/Documents/Developments/FantasyApp` (currently empty).

## Phase 1 — Core Foundation: Auth, Tenancy & Infrastructure · Week 1–2

**Goal:** stand up the running skeleton — the nginx gateway, RabbitMQ event bus, and the
**core-module** (`auth-service` + `permission-service`) generated by LAF and de-bugged — so that a
user can sign up, create a league (= tenant), and receive a JWT scoped to that league's schema, with
roles enforced. By the end of Phase 1 there is **no domain data yet** (no teams/weeks/scores — that
is Phase 2+), but the multi-tenant auth backbone every later service depends on is live and proven.

This phase is where the "Known LAF Generation Bugs" catalog (CLAUDE.md) first bites — `auth-service`
is generated by the same LAF agent that produced `tenant-api-net`, so **every bug in that list will
appear here**. Phase 1 is therefore as much about *fixing generation output* as writing new code.

**Reference to copy from:** `tenant-api-net/core-module/auth-service` and
`tenant-api-net/infrastructure` are the proven shape. When in doubt, diff against them.

### 1.0 Prerequisites & directory bootstrap

Before generating anything, lay down the monorepo skeleton (empty dirs + the shared library) so LAF
has somewhere to write and the gateway has something to route to.

```
fantasy-app/
├── core-module/                 # auth-service, permission-service live here
├── infrastructure/              # gateway, event-bus, adminer, (later) aws/
├── orchestrator/                # onboarding saga (1.7)
└── shared/                      # TenantModel base + CQRS/EDA contracts (copied from tenant-api-net)
```

- **Copy `shared/` wholesale from `tenant-api-net/shared`.** It holds `TenantModel`, the CQRS
  contract base classes (`SearchQueryContract`, `FetchQueryContract`, `BaseCommandContract`), the
  EDA event contracts, and `ContractFactory` — all of which the generated services `require` from
  `/var/www/shared` (mounted as a volume, see 1.9). The known-bug fixes for Bug 4 and Bug 8 are
  edits to files **in this shared lib**, so it must exist first.
- Create one Docker network shared by every container, e.g. `fantasy-app-net` (mirrors
  `synergy-erp-tenant-api-net`).

### 1.1 Registrar & port map (do this first — Bug 1 prevention)

`infrastructure/gateway/registrar.json` is the single source of truth for which internal port each
service listens on. **Assign a unique port per service up front** and use it everywhere (the
`php artisan serve --port=`, the compose `ports:`, and registrar). Defaulting to 9000 is LAF Bug 1
(→ 502 Bad Gateway on every request).

```json
{
  "infrastructure": [
    ["gateway", 80], ["event-bus", 5672], ["adminer", 8080], ["orchestrator-web", 1000]
  ],
  "core-module": [
    ["auth-service", 9001], ["permission-service", 9002]
  ],
  "league-module": [
    ["league-service", 9010], ["matchup-service", 9011], ["standings-service", 9012]
  ]
}
```

| Service | Internal port (registrar) | MySQL host port (dev) |
|---|---|---|
| auth-service | 9001 | 3301 |
| permission-service | 9002 | 3302 |
| league-service *(Phase 3)* | 9010 | 3310 |
| matchup-service *(Phase 4)* | 9011 | 3311 |
| standings-service *(Phase 4)* | 9012 | 3312 |

The nginx gateway strips the `/{service-name}/` URL prefix and proxies to `service:port`; Laravel
only ever sees `->prefix('api')`. Copy `infrastructure/gateway/{Dockerfile,entrypoint.sh,templates,
conf.d}` from `tenant-api-net` and edit `registrar.json` + the generated `conf.d` to the table above.

### 1.2 Event bus + supporting infra

- **RabbitMQ** as `event-bus` (host `EVENTBUS_HOST=event-bus`, port 5672). Copy
  `infrastructure/event-bus` from `tenant-api-net`. Every service gets an **inbox worker**
  (RabbitMQ consumer) and **outbox worker** (event publisher); the outbox/inbox tables live in each
  service's own schema (Bug 9, see 1.4).
- **adminer** on 8080 for eyeballing the MySQL schemas during bring-up.
- Dev credentials (RabbitMQ user/pass, MySQL root/laravel) live only in
  `docker-compose.development.yml` and `.env.development` — **never commit real secrets**; the
  Phase 0 commit hook enforces this.

### 1.3 Generate `auth-service` via LAF

The LAF agent (`laf-agent/agent.py`) is a remote, WebSocket-driven generator: you define the
service's **DB schema / manifest** in the LAF web tool, and the agent writes the Laravel service
(Models, Controllers, CQRS Contracts, `routes/api.php`, `manifest.json`, `docker/mysql-init/*.sql`,
Dockerfile) into `core-module/auth-service`. **`manifest.json` is ground truth** for every field
name/type — the frontend and all later cross-checks key off it.

**Schema definition input — auth-service models** (re-contextualized for fantasy-app; League =
Tenant). Mirror `tenant-api-net/core-module/auth-service` exactly except the tenant *means* a league:

| Model | Schema | Purpose |
|---|---|---|
| `Principal` | `accounts` | The actor identity (a person who owns a team / commissions a league). |
| `User` | `accounts` | Login credentials: `email`, `password` (bcrypt/argon2), `default_tenant_id`, `account_status_id`, `is_active`. |
| `Tenant` (= **League**) | `accounts` | `slug` (unique), `schema` (the per-league MySQL schema name), `tenant_profile_id`, `account_status_id`. **The `schema` column drives the JWT `schema` claim.** |
| `TenantProfile` | `accounts` | League display metadata (name, description, public/private flag). |
| `TenantPrincipal` | `accounts` | **Membership join** — which Principal belongs to which Tenant, with `starts_at`/`expires_at`/`revoked_at`. This is what every request validates the JWT `schema` claim against (don't trust the claim alone). |
| `PrincipalType` | `accounts` | user / service / api. |
| `AccountStatus` | `accounts` | active / suspended / pending taxonomy. |
| `WebClient`, `ApiClient` | `accounts` | OAuth/Sanctum client registrations. |
| `UserProfile` | `accounts` | name, avatar, etc. |
| `TransactionEvent` | `tenant` | Audit/event store row (shared infra). |

**Three schemas for auth-service** (auth is special, see CLAUDE.md "Database Setup"):
`auth-service` (own infra: inbox/outbox), `accounts` (cross-league auth — the tables above), and
`tenant` (per-tenant infra cache + `transaction_events`).

> ⚠️ Watch the LAF type bugs at the schema step (CLAUDE.md intro): wrong table names and bad
> data-type defs are the #1 source of generated bugs. Get `*_id` (BIGINT, internal FK) vs `*_puid`
> (CHAR(26), cross-service ULID) right in the manifest now.

### 1.4 Post-LAF audit + fix the known bugs (run `/laf-audit` FIRST)

Immediately after generation, **before touching the code**, run `/laf-audit`. Then apply each fix.
These are not optional — they are the difference between "boots" and "502 on every request":

| Bug | Symptom | Fix (concrete) |
|---|---|---|
| **1 — port 9000** | 502 on every request through gateway | Set `php artisan serve --port=9001` + `ports: ["9001:9001"]` in compose (1.1 table). |
| **2 — outbox entrypoint not mounted** | `entrypoint-outbox.sh: no such file or directory` | Add `- ./auth-service/docker/entrypoint-outbox.sh:/usr/local/bin/entrypoint-outbox.sh` to the outbox worker `volumes:`. |
| **3 — db_users.sql DROP/CREATE USER** | `Access denied for user 'laravel'@'%'` | Rewrite `db_users.sql` to **GRANT only** (`GRANT SELECT,INSERT,UPDATE,CREATE,INDEX,SHOW DATABASES ON *.* TO 'laravel'@'%'; FLUSH PRIVILEGES;`). Compose's `MYSQL_USER`/`MYSQL_PASSWORD` create the user. |
| **4 — BaseQueryContract not found** | `Class "...BaseQueryContract" not found` on search/fetch | In each `Search*QueryContract`/`Fetch*QueryContract`, change `extends BaseQueryContract` → `extends SearchQueryContract` / `extends FetchQueryContract` (from `shared/src/Contracts/Queries/`). |
| **8 — Search/Fetch wrong `rules()` + ctor args** | `TypeError: SearchQueryContract::__construct(): Argument #3 ($table)...` or search demands all fields | (a) Patch `shared/src/Controllers/Execution/ContractFactory.php` to call Search/Fetch contracts with correct signatures; (b) **delete the `rules()` override** from every `Search*`/`Fetch*` contract. |
| **9 — no `service_infra_models.sql`** | `Table 'auth-service.outbox_items' doesn't exist` | Create `docker/mysql-init/service_infra_models.sql` with `USE auth-service;` + `inbox_items`/`outbox_items` (copy from `tenant-api-net`). |
| **10 — broken command `rules()`** | HTML-entity-encoded `exists` rules; delete requires all fields | **Delete `rules()`** from every `Create*`/`Update*`/`Delete*` command contract (base `BaseCommandContract::rules()` is correct). **Keep `afterValidation()`.** |

Audit greps (from CLAUDE.md) to confirm each is clean:
```bash
grep -r "BaseQueryContract" core-module/auth-service/src/app/Contracts --include="*.php" -l | grep -v vendor   # Bug 4 → empty
grep -rl "extends SearchQueryContract\|extends FetchQueryContract" core-module/auth-service/src/app/Contracts --include="*.php" | grep -v vendor  # each: NO rules()
grep -rn "DROP USER\|CREATE USER" core-module/auth-service/docker/mysql-init/db_users.sql   # Bug 3 → empty
```

### 1.5 MySQL init files (raw SQL — NO Laravel migrations)

Populate `core-module/auth-service/docker/mysql-init/` (runs only on a fresh volume; apply manually
to a running container otherwise — see CLAUDE.md). Model on `tenant-api-net/.../auth-service/docker/
mysql-init`:

- `db_schemas.sql` — `CREATE DATABASE IF NOT EXISTS` for `auth-service`, `accounts`, `tenant`.
- `db_users.sql` — **GRANT only** (Bug 3).
- `accounts.sql` — the cross-league auth tables (`tenants`, `users`, `principals`, `tenant_principals`,
  `account_statuses`, `principal_types`, `tenant_profiles`, `user_profiles`, `web_clients`,
  `api_clients`). Note: `tenants.schema` column is what the JWT claim reads.
- `service_infra_models.sql` — `USE auth-service;` + `inbox_items` / `outbox_items` (Bug 9).
- `service_tenant_models.sql` — `USE tenant;` + `transaction_events` + caches.

**Seed taxonomy + a dev superadmin:** seed `account_statuses` (active/suspended/pending),
`principal_types` (user/service/api), and one bootstrap `User`+`Principal` so login can be tested
before the onboarding saga exists.

### 1.6 JWT / login / refresh wiring

- AuthN: `tymon/jwt-auth` (access + refresh) + Sanctum for API tokens (same as `tenant-api-net`).
- Endpoints (LAF special-cases these — they are **not** the `/{model}/{action}` shape):
  `POST /auth-service/login`, `POST /auth-service/logout` (a.k.a. `sign-out`),
  `POST /auth-service/refresh`, plus `POST /auth-service/me` and `GET /auth-service/validate`.
- **The `schema` claim:** on login, set the JWT `schema` claim from `accounts.tenants.schema` for the
  user's selected/`default_tenant_id` league. Every other service reads this claim to pick the
  tenant DB.
- **Membership validation middleware:** add middleware that, on every authenticated request,
  confirms the `schema` claim corresponds to a live `tenant_principals` row for the principal
  (not revoked/expired). A leaked or forged claim must not cross leagues. This is the security
  headline over v1.
- Bug 7 (by-design): if the `schema` claim names a DB that doesn't exist in a service yet, requests
  fail with "Schema does not exist." In Phase 1 the only schema is the dev seed league; the
  onboarding saga (1.7) generalizes this.

### 1.7 `permission-service` + roles (commissioner / member / viewer)

- Generate `permission-service` via LAF (registrar port 9002, MySQL 3302). Re-run `/laf-audit` and
  apply the same Bug 1–10 fixes. It owns `Role`, `Permission`, `RoleAssignment`
  (scoped per league).
- Seed three roles per league: **commissioner** (full CRUD on the league), **member** (read all,
  edit own team metadata only), **viewer** (read-only public pages, opt-in per league).
- Enforce via route middleware; every non-public route behind auth. The commissioner role is granted
  to the principal who created the league (set during onboarding, 1.8).

### 1.8 League onboarding saga (orchestrator)

Creating a league must, **atomically and before the first JWT is issued for it**:
1. Allocate a unique schema name (e.g. `league_<ulid>` or `league_<slug>`) and write the `Tenant`
   row in `accounts.tenants` with that `schema`.
2. **Create that schema in every service's MySQL** and run its `tenant_models.sql` (Phase 1: just
   auth-service's `tenant` infra; later services join as they come online).
3. Seed per-schema infra: `inbox_items`/`outbox_items`, `transaction_events`, caches (Bug 9 surface).
4. Create the commissioner `RoleAssignment` for the creating principal (1.7).
5. Only then issue the league-scoped JWT.

Implement as an **orchestrator saga** (`orchestrator/`) reacting to a `LeagueCreated` event over
RabbitMQ, with compensating steps on failure (drop the half-created schema, mark tenant `pending`).
In Phase 1 a thin synchronous version is acceptable to unblock testing, but the saga shape is the
target (CLAUDE.md "Onboarding"). Document it so Phase 2 services plug their schema-creation step in.

## Architecture Overview

Reuse the `tenant-api-net` shape exactly. **League = tenant** ⇒ each league has its own MySQL
schema; the JWT `schema` claim selects it on every request (LAF "Bug 7" is by-design tenancy).

```
fantasy-app/
├── core-module/
│   ├── auth-service/          # Principals, Users, Tenants(=Leagues), JWT, login/logout/refresh
│   └── permission-service/    # Roles & permissions (commissioner / member / viewer)
├── league-module/
│   ├── league-service/        # League, Season, Team(owner), TeamSeason membership
│   ├── matchup-service/       # Week, Matchup, Score  (the v1 "weeks" model)
│   └── standings-service/     # Computed standings, playoff brackets, all-time aggregates
├── import-module/             # (post-launch) CSV/JSON import + Sleeper sync workers
├── orchestrator/              # Sagas: recompute-standings-on-score-change, onboarding
├── infrastructure/            # nginx gateway + registrar.json, RabbitMQ, adminer, AWS configs
└── shared/                    # TenantModel base, CQRS/EDA contracts (from LAF)
```

**Conventions inherited from LAF (do not reinvent):**
- Route pattern: `POST /{service}/{model}/{action}` where action ∈
  `create|update|delete|restore|duplicate|batch-update|batch-delete|fetch|search`.
- Special auth routes: `POST /auth-service/login|logout|refresh`.
- `TenantModel` base fields: `id`, `public_id` (ULID, `CHAR(26)`), `public_ref`, `label`,
  `created_at`, soft-delete `deleted_at`, `created_by_principal`, `version` (optimistic lock).
- Each service: own MySQL container, inbox worker (RabbitMQ consumer), outbox worker (event
  publisher). nginx gateway strips the `/{service}/` prefix (Laravel uses `->prefix('api')`).
- **`manifest.json` per service is ground truth** for field names/types — frontend types must
  match it exactly.
- **No Laravel migrations** — schema via raw SQL init files in `docker/mysql-init/`.

## Domain Model (generalized from v1)

v1's `seasons.json` shape (`teams[]` with owner `name`, `state`, `playoff{made,rounds,pChampion}`,
`rChampion`; `weeks{}` → `matchups[]` with `team1/team1Score/...`, `status`; derived `standings`)
maps onto these tenant-scoped models:

| Service | Models | Notes |
|---|---|---|
| **league-service** | `League`, `Season`, `Team`, `TeamSeason` | `Team` = a franchise; an owner (`Principal`) can hold a team across seasons → enables "player/owner career stats" (v1 `PlayerStatsPage`). `TeamSeason` carries `state` (active/inactive/eliminated), `playoff_made`, `playoff_rounds`, `is_playoff_champion`, `is_regular_champion`. |
| **matchup-service** | `Week`, `Matchup` | `Matchup` = `team1_id`, `team1_score`, `team2_id`, `team2_score`, `label`, `status` (regular/playoff/toilet/out). BYE handling preserved. |
| **standings-service** | `Standing` (materialized), `PlayoffBracket` | Replaces v1's recompute-on-read (`recalculateStandings` in `server.js`). Recomputed via an **orchestrator saga** triggered by matchup score events, then stored — avoids v1's per-request recompute cost and concurrency bugs. |

Carry over v1 metrics: W/L/T, PF, PA, PFPG, PAPG, win %, place + prevPlace (rank delta Δ),
playoff buckets (playoff/toilet/out), champions. Fix v1 bugs in the port (e.g. `server.js:154`
`'name2' === 'BYE'` string-literal bug; magic "week ≥ 15 = playoff" → make playoff weeks
league-configurable).

## Authentication, Authorization & Security

This is the headline upgrade over v1 (which had none).

- **AuthN:** JWT via `tymon/jwt-auth` (access + refresh) + Sanctum for API tokens — same as
  `tenant-api-net`. `login/logout/refresh` endpoints. JWT carries the tenant `schema` claim.
- **AuthZ:** `permission-service` roles scoped per league:
  - **Commissioner** — full CRUD on their league (teams, weeks, scores, settings, invites).
  - **Member** — read all, edit own team metadata only.
  - **Viewer / public** — read-only public league pages (opt-in per league).
  - Enforce with route middleware; every non-public route behind auth.
- **Tenant isolation:** schema-per-league means a leaked query can't cross leagues; verify the
  JWT `schema` claim is validated against the principal's league membership on every request
  (don't trust the claim alone).
- **Onboarding:** new league ⇒ create its schema in every service's MySQL + seed infra tables
  (`inbox_items`/`outbox_items`, transaction cache) before issuing the first JWT (LAF "Bug 7/9").
  Automate via an orchestrator onboarding saga so it's not manual.
- **Hardening checklist:** HTTPS only; secure/HttpOnly/SameSite cookies; CORS locked to the
  known frontend origin (v1 `server.js:18` already scoped — keep that discipline); rate
  limiting on auth + write endpoints; input validation via LAF command contracts (strip the
  broken generated `rules()` — LAF Bug 8/10 — base contracts validate from manifest); CSRF on
  cookie-auth flows; bcrypt/argon2 password hashing; audit log via the existing
  `transaction_events` / event store; dependency scanning (Dependabot); secret management via
  env + the commit-blocking hook from Phase 0.

## Platform Sync — Research Findings (post-launch feature)

Verdict: **ship manual + CSV first, add Sleeper next.**

| Platform | Difficulty | Auth | Notes |
|---|---|---|---|
| **Sleeper** | **Easy** | None | Free read-only REST at `https://api.sleeper.app/v1/`, ~90 req/min/IP. Endpoints for leagues, rosters, users, matchups, playoffs, transactions, drafts, players. **Do this one first.** |
| **ESPN** | Medium | `espn_s2` + `SWID` cookies (private leagues) | Unofficial v3 API. Public leagues need no auth; private leagues need the user to paste two cookies (cannot be obtained programmatically). Mature JS/Python libs exist. |
| **Yahoo** | Hard | OAuth2 (3-legged) + refresh | Official API but requires app registration and full OAuth token lifecycle. Defer. |

**Plan:** Build an `import-module` with a pluggable `LeagueImporter` interface. Implement
`CsvImporter` + `SleeperImporter` first; the importer maps external rosters/matchups onto the
domain models above and runs as an outbox/inbox worker so a large import doesn't block the API.
ESPN/Yahoo are later adapters behind the same interface.
